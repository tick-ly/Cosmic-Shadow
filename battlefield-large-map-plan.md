# 战地风格大地图可执行计划

## 目标定义

本计划的目标不是一开始复刻完整《战地》规模，而是先做出“大规模战场流动感”。

首个可交付版本建议控制在以下范围：

- `16v16`
- `2km x 2km` 地图
- `5 个据点`
- `3 条主战线`
- `1 种地面载具`
- `无建筑破坏`
- `无飞机`
- `1 张灰盒地图`

这样可以先验证核心体验：远近交战、据点推进、载具介入、死亡重生后快速回到前线。

## 推荐技术路线

如果以浏览器或轻量原型为目标，推荐如下技术栈：

- 客户端：`Vite + TypeScript + Three.js`
- 物理与碰撞：`Rapier`
- HUD 与菜单：`DOM UI`
- 3D 资源格式：`GLB`
- 网络同步：`服务器权威`

架构必须从第一天开始分层：

- `simulation/`：游戏规则、队伍状态、据点状态、AI、可保存数据
- `render/`：场景、相机、动画、材质、粒子、可视化表现
- `physics/`：碰撞、刚体、射线检测、载具物理代理
- `ui/`：HUD、小地图、重生界面、设置菜单
- `diagnostics/`：FPS、draw call、网络延迟、地图分块调试

原则：

- 游戏状态不要直接挂在 Three.js 对象上
- 渲染层只负责“表现”，不负责“规则”
- HUD 和菜单默认放在 DOM，不塞进 WebGL

## 地图设计原则

先用灰盒验证战场流动，而不是先做高精度美术。

地图需要满足以下规则：

1. 出生后到第一次交火不超过 `30 秒`
2. 任意相邻据点间转场不超过 `60-90 秒`
3. 至少存在 `3 条主推进路线`
4. 每个据点都要同时支持近、中、远距离交战
5. 载具路线和步兵路线必须交叉，但不能完全重叠
6. 地图里必须存在天然阻隔，避免玩家全图互狙

建议把地图拆成 5 类功能区：

- `据点区`
- `开阔推进区`
- `天然阻隔区`，如山脊、河道、废墟带
- `载具主通道`
- `步兵绕后通道`

## 12 周执行计划

## 第 1-2 周：灰盒地图与战线验证

目标：做出能跑、能打、能形成主战线的白盒地图。

任务：

- 画 `2km x 2km` 俯视草图
- 标出 `5 个据点`、`2 个主出生区`、`3 条主推进线`
- 搭建基础地形高差、道路、掩体、山脊、壕沟
- 为每个据点布置近距离掩体、中距离推进线、远距离观察点
- 加入临时边界和回退区，避免玩家跑出战场

交付物：

- 可自由跑图的灰盒场景
- 顶视图设计图
- 据点与出生点分布图

验收标准：

- 玩家不看地图也能判断主战线方向
- 不会经常在无掩体开阔地暴毙
- 任意两条战线都具备不同交战节奏

## 第 3-4 周：步兵基础战斗闭环

目标：跑通最基础的单兵循环。

任务：

- 实现移动、跳跃、冲刺、蹲伏
- 实现开火、命中、受伤、死亡、重生
- 实现基本武器后坐与射线检测
- 实现准星、血量、弹药 HUD
- 实现临时音效与击中反馈

交付物：

- 单兵可完整游玩的灰盒战斗原型

验收标准：

- 玩家可以在据点周围持续作战
- 重生后能在较短时间内回到交战区
- 基础射击手感至少可调试、可迭代

## 第 5-6 周：据点系统与前线推进

目标：让地图从“能打”变成“有战局变化”。

任务：

- 实现据点占领、争夺、中立化逻辑
- 实现队伍控制分数或票数系统
- 实现基于据点控制状态的重生点切换
- 实现小地图和据点状态 UI
- 实现战线热区可视化调试

交付物：

- 一局可从开场推进到结束的占点模式

验收标准：

- 玩家能明显感受到前线在推进或回撤
- 据点控制会影响出生策略
- 战斗不会平均摊在整张地图上

## 第 7-8 周：载具接入

目标：让载具改变战场节奏，但不破坏步兵体验。

任务：

- 接入 `1 种轮式地面载具`
- 实现上下车、载具血量、基础移动
- 定义载具主路线和载具脆弱区
- 为步兵添加最基础的反载具手段
- 为据点附近设计反载具掩体和绕行路线

交付物：

- 载具可参与据点推进的战斗版本

验收标准：

- 载具能突破僵局，但不能单独统治整局
- 步兵仍然拥有绕后和反制空间

## 第 9-10 周：大地图性能与流式加载

目标：让大地图真正跑得动。

任务：

- 按 `128m` 或 `256m` 对地图进行 chunk 切分
- 实现静态物件 LOD
- 实现远景简化模型或 impostor
- 精简碰撞体，只保留近距离精细碰撞
- 统一纹理压缩、材质复用、网格合批策略
- 增加性能调试面板：FPS、draw call、显存、chunk 加载状态

交付物：

- 大地图性能优化版本

验收标准：

- 中配机器可稳定运行
- 高速移动时没有明显卡顿或大面积穿帮
- 资源加载峰值可控

## 第 11-12 周：联机与完整一局验证

目标：验证这张地图在多人战斗中是否成立。

任务：

- 接入服务器权威同步
- 同步玩家状态、命中、死亡、重生
- 同步据点状态和队伍积分
- 同步载具状态
- 增加最基础的延迟补偿与反作弊校验

交付物：

- `16 人以上` 的可玩联机测试版本

验收标准：

- 一整局流程可从开始打到结束
- 据点、载具、重生和计分表现一致
- 不因延迟导致核心玩法失真

## 核心技术清单

这些是“大地图”最关键的技术点，必须排进迭代：

- 地图分块加载
- 静态场景 LOD
- 简化碰撞与近远碰撞分层
- 远景假体或简化网格
- 稳定的资源命名、比例、pivot 规范
- 材质复用与纹理压缩
- 小地图与战线调试可视化
- 服务器权威同步

## 美术与场景制作规则

美术不要一开始追求写实，先保证可读性。

制作规范建议如下：

- 掩体做成模块化组件：沙袋、矮墙、废车、壕沟、断墙
- 每个资源必须统一比例和 pivot
- 所有节点命名稳定，便于脚本绑定
- 贴图分辨率按真实屏幕占比来定，不盲目上高
- 远景建筑和植被默认简化处理

优先顺序：

1. 地形可读性
2. 掩体布局
3. 远中近空间层次
4. 视觉风格统一
5. 细节装饰

## 风险控制

以下内容建议明确延后，不要在第一阶段接入：

- 建筑破坏
- 飞机或直升机
- 海量可进入建筑
- 超复杂弹道系统
- 高精度写实材质流程
- 超大规模 `64v64`

这些系统每加一个，复杂度都会成倍增加。

## 第一周立即执行任务

如果现在就开工，第一周按下面做：

1. 画一张顶视草图，确定据点、战线、出生点、道路
2. 搭建灰盒地形和掩体模块
3. 跑通步兵移动、射击、死亡、重生
4. 做最小 HUD：血量、弹药、据点状态
5. 做第一次内部测试，只看战线是否自然形成

## 每周评审问题

每周都用同一组问题检查方向是否跑偏：

1. 玩家是否会自然聚集到有趣的交战区域
2. 地图是否存在过长、过空、过无聊的移动区段
3. 据点攻防是否存在明显强点和弱点
4. 载具是否改变节奏而不是破坏平衡
5. 性能问题是否已经开始影响交战体验

## 成功标准

当以下条件成立时，这张地图就算进入“可继续扩展”的阶段：

- 单局节奏稳定
- 主战线清晰
- 据点推进有来有回
- 载具能改变局部战局
- 玩家死亡后能快速重返前线
- 地图在目标机器上性能稳定

## 下一步建议

写完这份计划后，最值得立刻继续拆的是两件事：

1. 产出一张 `2km x 2km` 的据点和主路线草图
2. 设计灰盒场景所需的模块化掩体清单
